## 背景與目標

直播穩定性改善討論（2026-07-11）拍板的第一個項目。核心價值：**網路再爛，比賽內容都完整留在手機裡**，事後可補傳 YouTube。已向 Boss 說明並確認的技術前提：

- 錄影資料在「編碼完成、還沒碰網路」的那一刻寫入手機，網路斷線／壅塞**不會**讓錄影卡頓或掉幀
- 錄影解析度**與直播相同**時共用同一顆編碼器（近零耗電），但網路壅塞自動降碼率會連帶降低錄影畫質
- 錄影解析度**與直播不同**時開第二顆獨立編碼器，錄影畫質全程恆定、與網路徹底脫鉤；代價是耗電發熱多約 10~15%、檔案較大（1080p 約 3.5~4.5 GB／小時）
- Boss 決定：做成「可開關＋可選解析度」，由使用者自行權衡

## 已查證的函式庫 API（RootEncoder 2.6.1，不得用猜的部分已全數確認）

- `Camera2Base.prepareVideo(width, height, fps, bitrate, iFrameInterval, rotation, profile, level, recordWidth, recordHeight, recordBitrate)`——公開多載；錄影解析度與直播不同時內部自動啟用第二顆編碼器（`differentRecordResolution`），**兩者長寬比必須相同**（本 APP 全部 16:9，無虞）；profile/level 傳 -1、iFrameInterval 傳 2 即等同現用的簡化多載
- `Camera2Base.startRecord(FileDescriptor, RecordController.Listener)`／`startRecord(String path, Listener)`——兩種都有；`stopRecord()` 收尾
- `RecordController.Listener`：`onStatusChange(Status)`（STARTED/STOPPED/RECORDING/PAUSED/RESUMED）＋ `onError(Exception)`（預設只寫 log，需覆寫）；**回呼來自背景執行緒，碰 UI 要 runOnUiThread**（比照 onNewBitrate 慣例）
- `setVideoBitrateOnFly` 只作用於直播那顆編碼器（原始碼第 963 行），錄影編碼器不受碼率自動調整影響——「獨立解析度=畫質恆定」的立論依據

## 功能包 ①：設定頁新增錄影選項

1. 新增開關「**同步錄影備份**」，預設**關**
2. 新增下拉「**錄影解析度**」，選項：
   - `與直播相同`（預設）——共用編碼器、近零耗電；說明文字註明「網路差時備份畫質會跟著直播降低」
   - `1280x720（720p）`
   - `1920x1080（1080p）`
3. 開關關閉時解析度下拉反灰不可選
4. 錄影碼率不另開選項（YAGNI）：`與直播相同`＝直接共用；720p 固定 4000 Kbps、1080p 固定 8000 Kbps（常數，加註解）
5. 新增「**錄影存檔位置**」（開關關閉時同樣反灰）：
   - `相簿`（預設）——走 MediaStore，影片直接出現在相簿
   - `自訂資料夾…`——按下開系統資料夾選取器（`ACTION_OPEN_DOCUMENT_TREE`），選定後
     `takePersistableUriPermission` 永久記住授權，設定頁顯示目前選的資料夾名稱；
     之後每場錄影在該資料夾內建檔（`DocumentFile.createFile`）→ `contentResolver.openFileDescriptor`
     → 交給同一個 `startRecord(fd, listener)`
   - 說明文字註明：自訂資料夾的影片不一定立刻出現在相簿，但檔案管理器一定找得到
6. `StreamPrefs` 新增 `record_enabled`（Boolean，預設 false）、`record_resolution`（String，預設「與直播相同」）、`record_save_mode`（String，預設「相簿」）、`record_tree_uri`（String，自訂資料夾的授權 Uri），比照現有 key 的存取寫法；**選項文字需與 strings.xml string-array 完全一致**（Spinner 還原機制，沿用現有慣例）

## 功能包 ②：直播主畫面錄影流程

1. `applyStreamSettingsAndStartPreview()` 內：錄影開啟且解析度≠直播時，改呼叫上述完整版 `prepareVideo` 多載帶入錄影參數；其餘情況維持現行呼叫（共用編碼器）
2. **開始**：`beginRtmpStreaming()` 內 `startStream()` 之後呼叫 `startRecord()`——錄影生命週期＝直播 session（開播即錄、收播即停）；就算 RTMP 一直連不上，錄影照跑（這正是備份的意義），直到使用者收播
3. **結束**：`stopLiveStream()` 內先 `stopRecord()` 再 `stopStream()`，確保 MP4 正常收尾
4. **存檔位置**（依設定頁「錄影存檔位置」）：
   - `相簿`＋API 29+（Android 10+）：`MediaStore.Video`＋`IS_PENDING` 流程取得 `FileDescriptor` 交給 `startRecord(fd, listener)`，收播後清除 pending——影片直接出現在相簿，YouTube APP 可直接選取上傳，全程免任何儲存權限
   - `相簿`＋API 26~28：退回 `getExternalFilesDir(DIRECTORY_MOVIES)` 路徑版 `startRecord(path,…)`（免權限；此區間裝置佔比極低，不為它引入權限流程）——`// ponytail:` 註解標明此天花板
   - `自訂資料夾`（不分 API 版本）：用已記住的授權 Uri 在該資料夾 `DocumentFile.createFile` 建 mp4 → `openFileDescriptor("rw")` → `startRecord(fd, listener)`；**開檔失敗（資料夾被刪、SD 卡拔除、授權被系統收回）→ 自動退回相簿模式續錄＋Toast 告知**，直播不受影響
   - 檔名：`籃球直播_yyyyMMdd_HHmmss.mp4`
5. **開播確認框**：錄影開啟時，訊息追加一行「將同步錄影（約 X.X GB／小時），目前可用空間 YY GB」；可用空間低於預估整場需求（以 2 小時計）時文字轉紅色警告，但**不阻擋開播**
6. **錄影中提示**：`tvLiveIndicator` 文字由「● 直播中」改為「● 直播中｜REC」（錄影啟用時），一眼確認備份有在跑
7. **錯誤處理（鐵律：錄影死掉絕不能拖垮直播）**：
   - `Listener.onError`／狀態異常 → runOnUiThread 跳 Toast「錄影備份失敗，直播不受影響：{原因}」，指示燈拿掉「｜REC」，直播照常
   - `startRecord` 拋 IOException（空間不足、MediaStore 失敗等）→ 同上處理，不中斷開播流程
8. 收播成功後 Toast 提示「錄影已儲存到相簿」（API 29+）或檔案路徑（舊版）

## 預期結果

- 設定頁開啟錄影後，每次開播自動同步錄影、收播自動存檔進相簿
- 網路中斷／壅塞期間，錄影檔完整無缺；選獨立解析度時畫質全程恆定
- 錄影任何環節失敗，直播完全不受影響，只跳提示
- 開播前看得到空間夠不夠，錄影中看得到 REC 有在跑

## 已知風險 / 注意事項

- 少數低階裝置不支援兩顆編碼器同時跑 1080p+720p——`prepareVideo` 回傳 false 時沿用現有的「串流規格失敗」Toast 提示，使用者改回「與直播相同」即可
- MediaStore `IS_PENDING` 若 App 中途被殺，會留下 pending 殘檔——收播與 onDestroy 都要清 pending；此為既知邊角，不做開機掃描回收（ponytail 註解標註）
- 2 小時 1080p 約 7~9 GB，空間警告只提醒不阻擋，錄到一半空間耗盡由 onError 路徑接手（直播不斷）
- 錄影共用編碼器模式下，BitrateAdapter 降碼率會反映在錄影檔——設定頁說明文字已載明，非 bug
- 自訂資料夾走 SAF 授權，換手機／清除 App 資料後授權消失，需重選一次資料夾（重選即可，非 bug）
